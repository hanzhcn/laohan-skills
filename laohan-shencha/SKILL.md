# Laohan Shencha（老韩深度审查）

## 核心原则

LLM 写技术文档时天然相信自己写的声明是对的。本 skill 的唯一目的：**找出并修正那些"看起来正确但实际是错的"声明。**

纯文字审查（表述矛盾、格式不统一）不是本 skill 的重点——那些靠文本比对就能发现。

## 五阶段审查流程

### 阶段 1：扫描 — 提取可验证声明

读取工作区所有相关文件，提取以下类型的声明：

| 声明类型 | 示例 | 验证方法 |
|---------|------|---------|
| 仓库 URL | `github.com/xxx/yyy` | GitHub API / git ls-remote |
| 版本号 | `PyTorch 2.7.1`, `v0.1.15` | PyPI / npm / GitHub releases |
| 参数值 | `blocks_to_swap=10`, `tile_size=128` | 官方文档 / README |
| 硬件需求 | `16GB VRAM`, `需要20GB` | 官方 README / 社区测试报告 |
| 兼容性 | "支持 Wan 2.2", "不兼容 X" | GitHub issues / discussions |
| 文件大小 | `~10GB`, `~37GB 总计` | HuggingFace / 实际文件 |
| 文件路径 | `wan_2.1_vae.safetensors` | HuggingFace API 文件列表（`/api/models/{repo}/tree/main`） |
| 安装步骤 | `pip install xxx`, `git clone yyy` | 实际执行 / 官方安装指南 |

**可执行脚本中的外部资源引用自动提升一个严重度等级。** 脚本（.ps1/.py/.sh）里的文件名、URL、仓库 ID 写错会直接导致运行失败，比文档中的同名错误严重得多。

输出：按文件分组的声明清单，标注每条的验证方法和优先级。

### 阶段 2：联网验证 — 逐条查证

对每条声明使用对应验证方法。**必须实际查询，不能凭训练知识判断。**

验证工具优先级：
1. `gh repo view` / `gh api` — GitHub 仓库可用性
2. Web search — 官方文档、社区报告
3. Context7 — 库/框架的最新文档
4. 直接 curl/请求 — API 端点、下载链接

**并行化：** 独立的验证项用多个 Agent 并行跑，不要串行。

### 阶段 3：交叉比对 — 文档间一致性

检查同一参数/描述在不同文件中的表述是否一致：

- 数值参数（节点数、VRAM、版本号）
- 仓库 URL 和安装命令
- 硬件需求描述
- 步骤/流程描述

### 阶段 4：修复 — 逐项修正

按严重度排序修复：

| 严重度 | 定义 | 示例 |
|-------|------|------|
| CRITICAL | 会导致运行失败 | 仓库不存在、命令语法错误 |
| HIGH | 会导致运行结果错误 | 参数值错误、VRAM 估算偏差大 |
| MEDIUM | 影响使用体验 | 版本号过时、描述不精确 |
| LOW | 文字描述问题 | 措辞不统一 |

### 阶段 5：出报告

输出格式：

```
## 审查报告

### 统计
- 总声明数：X
- 已验证：X
- 有问题：X（CRITICAL: X / HIGH: X / MEDIUM: X / LOW: X）
- 已修复：X

### 发现的问题
| 文件 | 行 | 问题 | 严重度 | 状态 |
|------|-----|------|--------|------|
| ... | ... | ... | ... | 已修/待定 |

### 仍需实际验证的项
（需要实际运行环境才能确认的项目）
```

## 关键注意事项

1. **不信任训练知识**：即使是"常识性"的参数值，也要联网验证。模型训练数据有时效性。
2. **验证仓库可用性**：`cubiq/ComfyUI_Face_Restore` 类问题（仓库被删/私有化）只有实际查才知道。
3. **参数值查官方源**：blocks_to_swap 的推荐值、VRAM 需求，以官方 README/文档为准，不以博客为准。
4. **跨文档同步**：修了一处参数，必须检查所有引用该参数的文件是否也需要更新。
5. **标注不确定项**：无法通过网络验证的（如需要实际硬件测试），在报告中明确标注。

## 适用项目类型

- AI 模型部署方案（ComfyUI、LoRA 训练、推理服务）
- 基础设施配置（Docker、K8s、CI/CD）
- 依赖版本密集的技术项目
- 多文档维护的项目（指南 + 脚本 + 工作流）
